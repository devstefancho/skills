# Test, Commit, Push, PR, Clean

<context>
!`git status`: 작업 상태
!`git worktree list`: 워크트리 목록
!`git log --oneline -10`: 최근 커밋
!`git branch --show-current`: 현재 브랜치
</context>

<instruction>

`$ARGUMENTS`에 `--skip 단계1,단계2,...` 가 있으면 해당 단계를 스킵한다. 사용 가능한 키: lint, test, push, pr, clean (예: `--skip push,pr,clean`)

1. 브랜치 체크 — 현재 브랜치가 디폴트 브랜치(main/master/develop)이면 작업을 중단하고, 워크트리/브랜치 생성을 안내
2. [lint] lint/format 체크
3. [test] 테스트 설정이 있으면 커버리지 100%로 맞추기, 없으면 스킵
4. 변경사항을 논리적 단위의 단일 커밋들로 정리. 아래 commit 템플릿 준수.
5. [push] git push
6. [pr] `gh pr create`로 PR 생성 (디폴트 브랜치 대상). 아래 PR 템플릿 준수.
7. [clean] `git worktree list` + `git branch --merged main`으로 머지 완료된 다른 워크트리가 있으면 `git worktree remove`로 제거 (현재 작업 워크트리는 유지)

</instruction>

<pr-template>

PR은 **한국어**로 작성한다. title은 기존 컨벤션(예: `feat(task-042): ...`)을 따르되 본문은 아래 섹션 구조로 작성한다.

## 기능 요약
이 변경이 **사용자/제품 관점**에서 무엇을 가능하게 하는지, 왜 필요한지 2–4줄로 서술. 코드 용어(함수명, 파일명) 금지. "어떤 기능이 추가/수정되는가", "어떤 문제를 해결하는가"에 집중.

예시:
> 사용자가 계정을 완전히 삭제할 수 있도록 GDPR 요건에 맞는 탈퇴 기능을 추가했다. 탈퇴 시 프로필, 아바타 이미지, 추천 기록, 로그인 세션까지 모두 즉시 제거되어 다른 기기에서도 자동 로그아웃된다.

## 구현 상세
기능을 구현하기 위해 **기술적으로 어떻게 했는지** 서술. 핵심 파일/함수 경로는 백틱으로 표기. 코드 수정 나열이 아니라 "왜 이 방식인지"를 드러내는 구현 결정 위주.

**영역별로 서브섹션을 나눠서 작성**한다. 변경이 닿은 영역만 포함하고, 해당 영역이 없으면 생략한다. 각 영역은 `### 백엔드`, `### 프론트엔드`, `### 인프라 / DevOps`, `### 데이터베이스 / 마이그레이션`, `### 테스트 하네스`, `### 공용 / 라이브러리` 등 자유롭게 선택.

**영역이 하나뿐이어도 서브섹션 헤더를 반드시 유지한다.** 헤더는 내용을 읽기 전에 "이 변경이 어느 영역에 속하는지"를 즉시 알려주는 라벨이다. 헤더가 없으면 리뷰어가 본문을 읽은 뒤에야 영역을 파악하게 되어, 기대하던 "읽기 전 컨텍스트"가 사라진다. 한 영역짜리 작은 PR이라도 예외 없이 헤더를 붙인다.

각 서브섹션은 불릿 2–5개. 전체 총합이 지나치게 길면 핵심 결정 위주로 압축.

예시 구조:

```
### 백엔드
- 엔드포인트 `onRequestDelete` (`packages/web/functions/api/users/me/index.ts`)
  추가 — 인증/CSRF를 DB·스토리지 접근 이전에 먼저 통과시켜 비용 큰 작업을 차단.
- D1 삭제는 자식 테이블 우선 명시 배치. D1이 커넥션 단위로
  `PRAGMA foreign_keys`를 보장하지 않아 CASCADE에 의존 불가 — 순서를 SSOT로 고정.
- 모든 기기 세션은 `deleteAllUserSessions`로 KV에서 즉시 무효화 (GDPR DoD).

### 프론트엔드
- 프로필 설정 페이지 (`apps/web/app/settings/profile.tsx`)에 탈퇴 버튼 추가,
  클릭 시 확인 다이얼로그 노출 후 API 호출.
- 성공 시 로컬 스토리지 세션 플러시 + 랜딩 페이지로 리다이렉트.

### 인프라 / DevOps
- `wrangler.toml`의 R2 바인딩에 쓰기 권한 추가 (기존 read-only).
- Cloudflare KV 네임스페이스 할당량 조정.
```

## 테스트
**QA/리뷰어가 어떤 동작이 검증되었는지 직관적으로 이해할 수 있도록** 사람의 언어로 서술. 테스트 파일명 나열 금지. "어떤 시나리오가, 어떤 기대 결과로 검증되었는가"를 사용자 동작 기준으로 기술.

예시:
- 로그인하지 않은 사용자가 탈퇴 API를 호출하면 401로 차단된다
- CSRF 토큰이 없거나 불일치하면 403으로 차단된다 (DB/스토리지 접근 전에 차단됨을 확인)
- 정상 탈퇴 시 프로필·추천 기록·아바타·다른 기기 세션이 모두 제거된다
- 동일 소셜 계정으로 재가입하면 과거 데이터 없이 완전히 새 계정이 된다

## 개발 중 확인 완료
이미 실행해서 통과한 검증을 체크된 상태로 기록. 리뷰어가 추가로 돌릴 필요가 없음을 명확히 한다.

- [x] lint 통과
- [x] 단위/통합 테스트 통과 (통과 개수 명시: 예 `322 passed`)
- [x] 타입 체크 통과 (해당 시)

## 배포 후 확인 필요
배포 후 사람이 수동으로 확인해야 할 시나리오가 있으면 **체크되지 않은 상태**로 나열. 없으면 이 섹션 자체를 생략한다. 리뷰어/QA가 실제 환경에서 수행해야 하는 E2E 시나리오에 한정.

- [ ] (예) 실제 소셜 로그인 → 탈퇴 → 재로그인 플로우를 프로덕션 환경에서 확인
- [ ] (예) 다른 기기에서 로그인 상태가 실제로 해제되는지 확인

작성 원칙:
- **"개발 중 확인 완료"와 "배포 후 확인 필요"는 절대 섞지 않는다.** 체크된 항목이 수동 테스트로 혼동되지 않도록 섹션을 분리한다.
- 배포 후 확인 항목이 없으면 섹션을 아예 빼서 리뷰어 혼란을 없앤다.
- 기능 요약에는 코드 용어를 쓰지 말고, 구현 상세에서는 반대로 구체적인 파일/함수명을 적극 사용한다.

</pr-template>

<commit-template>

커밋 메시지는 **한국어**로 작성한다. PR 본문과 달리 커밋은 개별 변경 단위이므로 간결함을 우선한다.

## 형식

```
<type>(<scope>): <제목>

<본문 — 선택>
```

- **type** (필수): `feat`, `fix`, `refactor`, `chore`, `docs`, `test`, `style`, `perf` 중 하나
- **scope** (선택): 변경이 집중된 영역. 모노레포라면 패키지명, 아니면 기능 영역 (예: `auth`, `profile-ui`). 애매하면 생략
- **제목**: 한국어, 50자 이내, 마침표 없이 끝. 명령형보다는 "~추가", "~수정" 같은 완료형이 읽기 좋음
- **제목과 본문 사이에 빈 줄 1개 필수**. git 표준이며 `git log`, `git shortlog` 등 도구가 이 빈 줄로 제목/본문을 분리함. 본문이 없으면 빈 줄도 생략 (제목만).
- **본문**: 제목만으로 "왜"가 자명하면 생략. 비자명한 결정·제약·트레이드오프가 있을 때만 추가. 72자 단위로 줄바꿈.

## 예시

자명한 변경 — 제목만:
```
feat(auth): GDPR 계정 삭제 API 추가
```

비자명한 결정 — 본문 포함:
```
refactor(d1): 계정 삭제 시 자식 테이블 명시적 배치로 변경

D1은 커넥션 단위로 PRAGMA foreign_keys를 보장하지 않아
CASCADE에 의존할 수 없다. 삭제 순서를 코드에서 SSOT로
고정해 환경별 차이를 제거한다.
```

## 원칙

- **한 커밋 = 한 논리적 단위**. 리팩터링과 기능 추가를 섞지 않는다. 섞였다면 커밋을 나눈다.
- **제목에 코드 용어(함수명 등) 남발 금지**. scope로 영역을 표시하고, 제목은 "무엇을 했는가"에 집중.
- **PR 본문과 중복 서술 피하기**. 커밋은 각 변경의 이유를 짧게 남기는 용도, 전체 맥락은 PR 본문이 담당.
- 타입 판단 기준: 사용자 기능 추가 → `feat`, 버그 수정 → `fix`, 동작 변경 없음 → `refactor`/`chore`/`docs`/`style`/`test`, 성능 개선 → `perf`.

</commit-template>
